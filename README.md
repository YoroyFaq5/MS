# 🃏 MafiaTracker

Полнофункциональное веб-приложение для клуба «Мафия»: учёт игр, рейтинги (ELO + сезонный рейтинг), турниры, сезоны, внутренняя экономика, Fantasy Draft, магазин кастомизации, система достижений и единый профиль игрока.

## Стек

- **Backend**: Python 3.11+ / Flask 3 / SQLAlchemy / Flask-Login
- **Database**: MySQL (драйвер PyMySQL, `mysql+pymysql://` в `DATABASE_URL`), схема управляется через Flask-Migrate/Alembic
- **Frontend**: Jinja2 + Bootstrap 5.3 (тёмная тема) + Bootstrap Icons + Inter font
- **Архитектура**: Blueprints (routes) → Service Layer (вся бизнес-логика) → SQLAlchemy models. Во view-функциях бизнес-логики нет — только вызовы сервисов и рендеринг.

## Быстрый старт

```bash
# 1. Клонировать / распаковать проект
cd MS

# 2. Создать виртуальное окружение
python -m venv .venv
source .venv/bin/activate        # Linux/macOS
# .venv\Scripts\activate         # Windows

# 3. Установить зависимости
pip install -r requirements.txt

# 4. Создать базу данных в MySQL (ВАЖНО: именно utf8mb4 — приложение
#    русскоязычное и хранит эмодзи в некоторых текстовых полях; обычный
#    "utf8" в MySQL не полный UTF-8 и не годится)
mysql -u root -p -e "CREATE DATABASE mafia_style CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# 5. Настроить окружение
cp .env.example .env
# Обязательно отредактировать DATABASE_URL (реальные логин/пароль/хост)
# и SECRET_KEY — без DATABASE_URL приложение не запустится

# 6. Применить миграции (создаёт все таблицы через Alembic)
flask --app run.py db upgrade

# 7. (Опционально) Заполнить тестовыми данными
flask --app run.py seed-players        # 10 тестовых игроков
flask --app run.py seed-shop           # примеры товаров магазина
flask --app run.py seed-achievements   # примеры достижений (все 9 категорий)

# 8. Запустить
python run.py
```

Откройте браузер: **http://127.0.0.1:5000**

Первый пользователь регистрируется через `/auth/register`. Права администратора выдаются вручную через БД или другим уже существующим админом на странице `/auth/admin/users`.

### Деплой на PythonAnywhere

MySQL на PythonAnywhere доступен только изнутри их инфраструктуры (в
Bash-консоли или из запущенного веб-приложения) — с локальной машины
подключиться не получится. Реальный хост/имя БД смотрите во вкладке
**Databases** своего аккаунта, но по конвенции PythonAnywhere это обычно:

```
DATABASE_URL=mysql+pymysql://<username>:<пароль>@<username>.mysql.pythonanywhere-services.com/<username>$mafia_style?charset=utf8mb4
```

(обратите внимание на `$` в имени базы — так PythonAnywhere изолирует
базы разных аккаунтов на одном сервере). `flask db upgrade` нужно
запускать в Bash-консоли PythonAnywhere, где переменная `DATABASE_URL`
уже указывает на их MySQL.

## Архитектура

**Service Layer.** Каждая предметная область (economy, tournaments, fantasy, shop, achievements…) — отдельный класс-сервис из статических методов. Каждый метод-мутатор возвращает небольшой DTO-результат (`ok: bool`, `message: str`, `data`), а не бросает исключения как основной механизм управления потоком. Views только вызывают сервисы, показывают `message` через `flash()` и делают `redirect`.

**Orchestrators.** Завершение игры или турнира — это цепочка вызовов нескольких сервисов в строго определённом порядке. Эта последовательность вынесена в `PostGameOrchestrator` / `PostTournamentOrchestrator`, чтобы views не знали о внутреннем порядке операций:

```
PostGameOrchestrator.run(game):
  1. RatingService.apply_base_scores_to_game()
  2. EloEngine.apply_match()                (если игра рейтинговая)
  3. SeasonService.resolve_season_for_game() (привязка к сезону)
  4. EconomyService.apply_rewards_after_game()
  5. commit()
  6. AchievementService.check_after_game()

PostTournamentOrchestrator.run(tournament):
  1. FantasyService.lock_drafts_for_tournament()
  2. FantasyService.score_tournament()        (+ выплата банка)
  3. EconomyService.apply_tournament_rewards()
  4. SeasonService.close_expired_seasons()
  5. AchievementService.check_after_tournament()
```

**RBAC.** Единая точка принятия решений — `PermissionService.can(user, Permission.X)`. Ни во views, ни в других сервисах проверки прав не дублируются.

**Права предметов и монет.** Все изменения баланса монет — неизменяемый журнал (`CoinTransaction`), только `INSERT`. GG-бонусы и покупки — то же самое: audit trail, никаких «тихих» `UPDATE` баланса без записи причины.

## Функциональность

### 1. Игроки и пользователи (RBAC)

- **`Player`** — карточка участника клуба (ник, настоящее имя, аватар, био, ELO, монеты). Может существовать без учётной записи (`User`).
- **`User`** — учётная запись для входа (логин/пароль через `werkzeug`, задел под OAuth). Может существовать без привязанного `Player` (админ без игрового профиля).
- Связь `User ↔ Player` — необязательная 1:1, управляется на странице `/auth/link-player` (не путает игровую статистику с логином — при отвязке/повторной привязке история игрока сохраняется).
- **Роли доступа**: `admin` / обычный пользователь / анонимный посетитель. Полная матрица — [ниже](#права-доступа-permission-matrix).
- Регистрация, вход, смена пароля, деактивация/повышение пользователей администратором — `/auth/*`.

### 2. Игры

- Игра — 10 слотов (мест), каждому назначена роль:

  | Роль | Кол-во | Сторона |
  |------|--------|---------|
  | Мирный житель | 6 | Город |
  | Шериф | 1 | Город |
  | Мафия | 2 | Мафия |
  | Дон | 1 | Мафия |

- **Базовые очки**: 1.0 за победу своей стороны, 0.0 за поражение (одинаково для всех ролей — простая и предсказуемая шкала).
- **Бонусные баллы** (`bonus_score`) — произвольные, вводятся администратором вручную при завершении игры.
- **ПУ («Первый убиенный»)** — если игрок первым выбывает ночью без промаха, он называет 3 подозреваемых; количество угаданных мафий (0–3) даёт бонус `+0.0 / +0.1 / +0.3 / +0.6` к очкам и учитывается как «особое событие» в формуле ELO.
- **Quality score** (`-1.0…+1.0`) — необязательная субъективная оценка вклада игрока (выставляется судьёй/админом), слегка влияет и на очки, и на ELO.
- Игры могут быть привязаны к турниру/этапу (`tournament_id`, `stage_id`) и наследуют его флаг «рейтинговая» (`is_ranked`).
- Завершение игры запускает `PostGameOrchestrator` — пересчёт очков, ELO, привязка к сезону, начисление монет, проверка достижений — одной транзакцией.

### 3. Рейтинговая система

Три независимых, никогда не смешивающихся друг с другом рейтинга:

**a) Общий рейтинг (`RatingService.compute_all_ratings`)** — сумма очков по всем рейтинговым завершённым играм, сортировка по сумме ↓, при равенстве — по среднему баллу ↓.

**b) ELO (`EloEngine`)** — рассчитывается после каждой рейтинговой игры по формуле:

```
Δμᵢ = α·(Rᵢ−Eᵢ)·kᵢ·uᵢ  +  α·sᵢ·K  +  λ·bᵢ
```

| Компонент | Значение |
|---|---|
| `Eᵢ` | ожидаемый результат — логистическая функция разницы среднего ELO команд |
| `Rᵢ` | фактический результат (победа/поражение + небольшая поправка на `quality_score`) |
| `kᵢ` | множитель роли (Шериф ×1.25, Дон ×1.20, Мафия ×1.05, Мирный ×1.00 — решения с более высоким скилл-потолком весят больше) |
| `uᵢ` | коэффициент неопределённости — новички (< 15 игр) двигаются быстрее (×1.6), ветераны (> 50 / > 100 игр) — медленнее (×0.85 / ×0.65), защита от инфляции рейтинга |
| `sᵢ` | quality score судьи, вес `K = 6.0` |
| `bᵢ` | бонус за особые события (успешный ПУ), взвешен `λ = 0.5`, с анти-абьюз потолком |

Детерминированный, чистые функции — легко тестируется в изоляции.

**c) Сезонный рейтинг (`SeasonRatingEngine`)** — полностью отдельная формула, никогда не читает и не пишет ELO:

```
SeasonRating = (TotalPoints × WR%) + (GG × 0.2)
```

где `TotalPoints` — сумма очков за сезон, `WR%` — процент побед за сезон, `GG` — сумма ручных бонусов администратора за этот же сезон (`GGService`, только `INSERT`, полный audit trail, дневной лимит на админа).

Также доступны: рейтинг турнира/этапа, командный рейтинг, годовой рейтинг (агрегация 6 сезонов).

### 4. Турниры

- Два типа: **индивидуальный** и **командный** (`Team` / `TeamPlayer`, игрок закреплён максимум за одной командой турнира).
- Жизненный цикл: `pending → active → finished`.
- Опциональные этапы (`TournamentStage`): «Основной этап» → «Финал», с **cutoff**-механикой — топ-N игроков основного этапа (по рейтингу этапа) автоматически проходят в финал (`TournamentService.run_cutoff`, идемпотентно).
- Массовая генерация игр (`generate_games`) — жадный алгоритм распределения по местам, минимизирующий повторное сидение одного игрока на одном месте.
- Флаг `is_ranked` турнира определяет, влияют ли его игры на общий рейтинг/ELO.
- По завершении турнира — `PostTournamentOrchestrator`: подсчёт Fantasy, денежные призы, проверка достижений.

### 5. Сезоны

- Год автоматически делится на **6 сезонов по 2 месяца** (Янв–Фев … Ноя–Дек), создаются лениво (`ensure_year_exists`) при старте приложения и при первой необходимости.
- Игра автоматически привязывается к сезону по дате (`resolve_season_for_game`), идемпотентно.
- Сезон закрывается автоматически по истечении периода (`close_expired_seasons`): считается победитель по `SeasonRating`; при **ничьей** — статус `WAITING_TIEBREAK`, администратор выбирает победителя вручную (`resolve_tiebreak`).
- Победитель каждого сезона автоматически регистрируется в турнире **«Стол года»**; после завершения всех 6 сезонов формируется финальный турнир года.
- Закрытие сезона запускает выплату денежных призов и проверку сезонных достижений.

### 6. Экономика (монеты)

- Внутренняя валюта на `Player.coins`, каждое изменение — неизменяемая запись `CoinTransaction` (сумма, причина, тип источника, баланс после операции).
- Источники начислений: игра (участие + победа), турнир (участие + место), сезон (победа / топ-3 / топ-10), Fantasy (банк), ручная корректировка администратором.
- **Анти-абьюз**: суточный лимит заработка с игр на игрока (`DAILY_GAME_REWARD_CAP`), обязательная причина ≥10 символов для ручных корректировок администратора.
- Настройки Fantasy (стоимость входа, доля банка за 1-е/2-е место) редактируются администратором в реальном времени (`/auth/admin/economy`) без деплоя.

### 7. Fantasy Draft

- Пользователь (не игрок этого турнира) выбирает 3–5 игроков турнира, в котором сам не участвует (лимит пиков растёт с числом участников турнира).
- Вход платный — списывается сразу при создании драфта, сумма «замораживается» на драфте (`entry_cost_paid`), так что последующее изменение цены админом не влияет на уже идущие турниры.
- После завершения турнира очки драфта = сумма реальных турнирных очков выбранных игроков (не таблица мест).
- Призовой банк = сумма всех входных взносов; 1-е/2-е место в лидерборде драфтов забирают свою долю банка — по-настоящему пирамида «игроки скидываются — лучшие прогнозисты выигрывают», без эмиссии новых монет.

### 8. Магазин

- **`ShopItem`** — товар: название, описание, цена, редкость (`common / rare / epic / legendary`), картинка, категория + подкатегория (равно «слот» экипировки).
- Три категории:
  - **Кастомизация профиля** (`profile_customization`) — рамки, фоны, оформление статистики, декоративные элементы;
  - **Никнейм** (`nickname`) — цвет, градиент, анимация, префикс, суффикс (влияет **только на отображение**, не на `Player.name`/логин);
  - **Физические товары** (`physical`) — кружки, футболки, сертификаты, билеты на турнир и т.п. Покупка лишь списывает монеты, выдача — вручную администратором вне приложения.
- Слот экипировки = `category` + `subcategory`. В одном слоте одновременно экипирован максимум один предмет — экипировка нового автоматически снимает предыдущий того же слота. Новые слоты (подкатегории) добавляются **без изменения кода** сервиса (data-driven).
- По умолчанию товар можно купить один раз на игрока (`is_unique_purchase`); для физических товаров это ограничение снимается администратором.
- Администрирование (`/admin/shop`): создание / редактирование / отключение (soft-delete) / ручная выдача товара игроку с указанием причины.

### 9. Инвентарь

- `InventoryItem` — купленный/выданный экземпляр товара у конкретного игрока (дата покупки, уплаченная цена — снимок на момент покупки, статус экипировки, источник: покупка / выдача админом).
- Предмет не исчезает после покупки — хранится в инвентаре навсегда, можно экипировать/снять в любой момент (`/inventory`).
- Физические товары никогда не экипируются — это чисто «квитанция» о покупке.

### 10. Достижения

- **`Achievement`** — презентационные данные (название, описание, иконка, редкость, категория, признак «скрытое»), связаны с правилом разблокировки через уникальный `code`.
- 9 категорий: Игры, Победы, Рейтинг, Турниры, Сезоны, Fantasy, Экономика, Социальные, Специальные.
- **Реестр правил** (`achievement_rules.py`) — открытая для расширения архитектура: новое достижение = одна запись в реестре (код + условие) + одна строка в БД с тем же кодом. Диспетчер (`AchievementService`) никогда не меняется.
- Проверка запускается автоматически по 4 триггерам: после игры, после турнира, после сезона, после покупки в магазине (плюс разовая ручная разблокировка при привязке аккаунта и полностью ручная выдача администратором для «Специальных» достижений).
- Скрытые достижения показываются как «???» до момента получения.
- Разблокировка идемпотентна (защита от повторного срабатывания хуков).
- Игрок может закрепить **до 3** достижений — они отображаются рядом с ником в профиле.

### 11. Профиль игрока

Единая страница `/profile/<player_id>` (доступна для просмотра любого игрока, не только своего), заменившая старые `/players/<id>/stats` и `/auth/profile`:

- **Основной блок** (дёшево, без тяжёлых запросов): аватар, отображаемый ник (с применённым оформлением), био, дата регистрации, монеты, место в общем рейтинге, закреплённые достижения.
- **`/profile/<id>/statistics`** — полная статистика:
  - всего игр / побед / поражений / WR / общий рейтинг / ELO / турниры / победы в турнирах / сезоны / заработано и потрачено монет;
  - игровая аналитика: любимая / лучшая / худшая роль, лучшая серия побед, худшая серия поражений, текущая серия, средние/лучшие/худшие баллы за игру, % побед по каждой роли;
  - **напарники**: лучший и худший напарник (по числу совместных побед/поражений), самый частый союзник и соперник, лучший и худший WR против конкретного оппонента (с минимальным порогом совместных игр против шума) — вычисляется за **2 запроса** независимо от количества сыгранных игр;
  - турниры: количество, лучший результат, число финалов, победы, подиумы;
  - Fantasy: количество драфтов, лучший драфт, среднее место, суммарный fantasy-рейтинг.
- **`/profile/<id>/achievements`** — все достижения: полученные / неполученные / скрытые, процент выполнения, закрепление до 3 штук.
- **`/profile/<id>/customization`** — просмотр текущего оформления (только чтение); управление экипировкой — на странице `/inventory`.

Прицельно спроектировано так, чтобы на главную страницу профиля не выполнялись десятки SQL-запросов — тяжёлые вычисления вынесены на отдельные подстраницы и загружаются только при переходе на них.

## CLI-команды

```bash
flask --app run.py db upgrade         # применить все миграции (основной способ создать/обновить схему)
flask --app run.py init-db            # альтернатива для быстрого локального бутстрапа: db.create_all() напрямую, без Alembic
```

## Переменные окружения

| Переменная | По умолчанию | Описание |
|-----------|-------------|---------|
| `SECRET_KEY` | dev-key | Секрет Flask (обязательно изменить в prod) |

База данных должна быть создана с `CHARACTER SET utf8mb4` (см. «Быстрый
старт») — обычный `utf8` в MySQL не полный UTF-8 и не хранит эмодзи,
которые встречаются в тексте приложения.

## Разработка

Схема БД управляется через **Flask-Migrate / Alembic** (`migrations/`).
После изменения моделей:

```bash
flask --app run.py db migrate -m "Описание изменения"   # сгенерировать миграцию
# проверить сгенерированный файл в migrations/versions/ — autogenerate
# не всегда всё видит правильно (например, переименования колонок)
flask --app run.py db upgrade                            # применить
```

`db.create_all()` (команда `init-db`) — резервный способ для быстрого
локального бутстрапа/тестов, но не заменяет миграции: он не умеет менять
уже существующие таблицы, только создавать отсутствующие. Три старых
точечных скрипта в корне проекта (`migrate_fantasy_economy.py`,
`migrate_elo_history.py`, `migrate_gifting.py`) относятся к более раннему,
до-Alembic периоду проекта — для новой базы, поднятой через
`flask db upgrade`, они не нужны (все их изменения уже входят в схему).

Принципы, которых стоит придерживаться при расширении:
- Вся бизнес-логика — в `app/services/*`, views остаются тонкими.
- Любая проверка прав — только через `PermissionService`.
- Любое изменение баланса монет — только через `EconomyService` (неизменяемый журнал).
- Новые достижения добавляются через `achievement_rules.py` (Open/Closed) — без изменения `AchievementService`.
- Новые типы кастомизации магазина — просто новая `subcategory`, без изменения `ShopService`.
